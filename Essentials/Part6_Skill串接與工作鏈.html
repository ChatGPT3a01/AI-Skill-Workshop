Part 6 · Skill 串接與工作鏈

讓 Skill 分工合作

把多個 Skill 串成工作鏈，A 做完換 B 做，打造自動化流程

🔗 Skill 串接 ⛓️ 工作鏈 ⚙️ 自動化 📤 輸入/輸出對齊

🤔 為什麼要串接 Skill？

單一 Skill 只能做一件事，但真實工作常常需要多個步驟協作完成。

🚫 Before：單一 Skill

  • ❌ 一次只做一件事
  • ❌ 手動貼上下一步的輸入
  • ❌ 人工檢查品質
  • ❌ 效率低，容易出錯

✅ After：Skill 工作鏈

  • ✅ 自動串接多任務
  • ✅ AI 自動傳遞結果
  • ✅ 品質層層把關
  • ✅ 一鍵執行，高效可靠
💡 類比
單一 Skill 像一位專員；Skill 工作鏈像一條產線 — 每個站點負責一個環節，自動傳遞給下一站。

🍜 一道菜 vs 一家餐廳

把單一 Skill 和 Skill 工作鏈想成「一道菜」與「一家餐廳」的關係：

🍳 一道菜（單一 Skill）

  • 👨‍🍳 一個廚師做一道菜
  • 📋 只有一份食譜
  • ✅ 好吃但不夠
  • 📧 例如：只回覆 Email

🏨 一家餐廳（Skill 工作鏈）

  • 👨‍🍳 多個廚師各司其職
  • 📋 多份食譜串接成套餐
  • ✅ 完整體驗，物超所值
  • 📧 例如：分析需求→撰寫提案→審查品質
✅ 重點
多個專精 Skill 串接起來，比一個巨大的 Skill 更靈活、更可靠、更容易維護！

✂️ 拆解原則：怎麼把流程切成多個 Skill？

不是隨便切就好，要遵守以下四個拆解原則：

🎯

1. 單一職責

每個 Skill 只做一件事。「分析需求」和「撰寫提案」是不同的 Skill。

📤

2. 明確界面

每個 Skill 要清楚定義「輸入什麼」和「輸出什麼」，讓 AI 能正確傳遞。

🧪

3. 獨立可測試

每個 Skill 單獨也能使用。串接是加分項，不是必要條件。

🔄

4. 可重複使用

好的拆解讓單一 Skill 可以在多條工作鏈中重複使用。

💡 判斷基準
如果你說不清楚這個 Skill 「到底在做什麼」，表示它太大了，應該繼續拆解！

🏗️ 串接架構圖解

以「客戶來信 → 完整提案書」為例，三個獨立 Skill 串接成一條工作鏈：

📥 客戶來信 原始需求文件
⬇️
🔍 need-analyzer 📤 輸出：需求分析摘要
⬇️ 分析摘要
📝 proposal-drafter 📤 輸出：提案草稿
⬇️ 提案草稿
🔎 proposal-reviewer 📤 輸出：審查報告 + 修正版
⬇️
📄 完整提案書 可直接提交客戶
✅ 關鍵設計
每個 Skill 的輸出剛好是下一個 Skill 的輸入！這就是串接成功的核心。

🔍 Skill 1：need-analyzer 需求分析

第一個 Skill 負責分析客戶來信，提取關鍵需求並分類。

SKILL.md 核心內容

---
name: need-analyzer
description: >
  分析客戶需求文件或來信，提取關鍵需求並分類。
  當使用者提供客戶的需求描述時使用此技能。
---

# 客戶需求分析器

## 執行步驟
### Step 1：需求識別
- 識別所有明確提出的需求項目
- 識別隱含的（未直接說明但可推斷的）需求

### Step 2：需求分類
- 核心需求：客戶明確要求，必須滿足
- 期望需求：希望有但非必要
- 潛在需求：AI 建議的加分項

### Step 3：風險標記
- Check if 需求涉及第三方整合 → 標記「需協調」
- Check if 時程低於 2 週 → 標記「時程風險」

### Step 4：產出結構化摘要
- MUST 包含核心需求表格（含優先級）
- NEVER 遺漏客戶明確提到的任何需求
💡 輸出格式
need-analyzer 的輸出是結構化的需求摘要，包含需求表格、風險標記、優先級等資訊，供 proposal-drafter 接收使用。

📝 Skill 2：proposal-drafter 提案撰寫

第二個 Skill 負責根據需求分析摘要撰寫客戶提案書草稿。

SKILL.md 核心內容

---
name: proposal-drafter
description: >
  根據需求分析摘要撰寫客戶提案書草稿。
  當使用者提供需求分析結果或要求撰寫提案書時使用。
---

# 客戶提案撰寫器

## 前置條件
- MUST 確認已有需求分析摘要（來自 need-analyzer）
- Check if 使用者直接提供原始需求 → 建議先執行 need-analyzer
- Ask user 提供公司名稱和方案定價範圍

## 執行步驟
### Step 1：撰寫提案架構
- 依照 @assets/proposal-template.md 範本

### Step 2：填充內容
- MUST 針對每個核心需求提出具體解決方案
- MUST 包含時程規劃與報價概估
- NEVER 承諾未確認的技術細節
💡 輸入與輸出
輸入：need-analyzer 的需求摘要 + 公司資訊
輸出：完整的提案書草稿（含架構、方案、時程、報價）

🔎 Skill 3：proposal-reviewer 品質審查

第三個 Skill 負責審查提案書草稿的品質，找出問題並改善。

審查項目

面向 檢查重點 評分
完整性 是否涵蓋所有核心需求？ /10
說服力 是否有明確的價值主張？ /10
可行性 時程和預算是否合理？ /10
格式 是否符合專業商務文件格式？ /10

SKILL.md 核心

## 執行步驟
- 逐項檢查四個面向
- MUST 列出所有問題（按嚴重程度排序）
- MUST 提供每個問題的具體修改建議
- 自動修正明確的錯誤（錯字、格式不一致）
- Ask user 確認需要人工判斷的重大修改
- NEVER 降低提案書的專業水準
💡 輸入與輸出
輸入：proposal-drafter 的提案草稿
輸出：審查報告 + 修正後的最終版提案書

🔑 串接的關鍵設計

要讓多個 Skill 順利串接，需要遵守以下四個關鍵原則：

📤

1. 輸入/輸出對齊

A 的輸出格式必須符合 B 的輸入格式。在 SKILL.md 的 description 中明確寫出「接收什麼」和「輸出什麼」。

📋

2. 明確傳遞資訊

每個 Skill 的 description 要說明「我需要什麼輸入」和「我會輸出什麼」，讓 AI 能正確傳遞。

🧪

3. 獨立可測試

每個 Skill 單獨也能使用。串接是加分項，不是必要條件。先確保單一 Skill 正常再串接。

⚠️

4. 錯誤處理

中間環節出錯時要有 fallback 機制。例如：「Check if 缺少需求摘要 → Ask user 補充」。

💡 設計訣竅
在 SKILL.md 的 description 中明確寫出：「此 Skill 接收 [需求分析摘要]，輸出 [提案草稿]」，AI 就能正確串接。

📝 Part 6 重點回顧

在這個單元中，我們學會了如何把多個 Skill 串接成工作鏈，打造自動化流程。

🔗

串接 = 分工合作

多個專精 Skill 串接起來，比一個巨大的 Skill 更靈活更可靠。

📤

輸入/輸出對齊

A 的輸出 = B 的輸入，這是串接成功的核心關鍵。

🧪

每個 Skill 獨立可測試

串接是加分項，每個 Skill 單獨都能使用，維護更容易。

🎯

實戰應用廣泛

從客戶服務到內容產出，Skill 工作鏈可以應用在各種場景。
