Part 4 · 在 Antigravity 執行 Skill

讓你的 Skill 跑起來！

登入 Google Antigravity 平台，部署你的 SKILL.md，實際測試並驗證執行結果

Antigravity 免費平台 實測執行

🚀 什麼是 Antigravity？

Antigravity 是 Google 推出的免費 AI Agent 開發平台，內建 Claude 模型，完整支援 Agent Skills 功能。不需要 API Key、不需要安裝任何軟體，只要有 Google 帳號就能使用。

🆓 完全免費

不需要 API Key，不需要付費訂閱。零成本開始你的 AI Skill 之旅，適合所有想入門的學員。

🖥️ 圖形化 IDE

網頁版開發環境，內建檔案管理器、終端機、AI 對話面板。打開瀏覽器就能開始工作，不需要安裝任何東西。

📁 原生支援 Skills

完整支援 .agent/skills/ 資料夾結構，AI 會自動掃描並載入你的 Skill 檔案，不需要額外設定。

🔄 即時測試

修改 SKILL.md 後，下一次對話就會自動載入最新版本。開發、測試、迭代一條龍完成。

💡 平台網址：antigravity.dev -- 用 Google 帳號登入即可開始使用。建議使用 Chrome 瀏覽器以獲得最佳體驗。

🔑 登入 Antigravity

開啟瀏覽器，前往 antigravity.dev，用你的 Google 帳號登入。登入後你會看到 Antigravity 的儀表板（Dashboard），這裡顯示你的所有專案。

登入步驟

  1. 前往 antigravity.dev
  2. 點擊「Sign in with Google」按鈕
  3. 選擇你的 Google 帳號並授權
  4. 登入成功後，進入儀表板
  5. 點擊「New Project」建立新專案，或開啟既有專案
✅ 登入成功後：你會看到一個類似 VS Code 的介面，這就是 Antigravity 的 IDE 環境。左邊是檔案管理器，中間是編輯器，右邊是 AI 對話面板。
⚠️ 注意：如果你是第一次使用 Antigravity，建議先建立一個空白專案來練習。之後的操作都會在這個專案中進行。

🖥️ IDE 介面總覽

進入專案後，你會看到一個完整的 IDE 開發環境。這個介面分為三大區域，每個區域都有特定的功能：

📂 左側面板：檔案管理器

顯示專案的所有檔案和資料夾。你可以在這裡新增、刪除、重新命名檔案，也可以直接拖放上傳檔案。

📝 中央區域：程式碼編輯器

類似 VS Code 的文字編輯區，可以直接編輯 SKILL.md 和其他檔案。支援語法高亮、自動補全等功能。

💬 右側面板：AI 對話

與 AI 互動的對話視窗。在這裡輸入你的需求，AI 會自動偵測可用的 Skills 並按照規則執行。

💡 操作提示：你可以拖動面板之間的分隔線來調整各區域的大小。如果 AI 對話面板沒有顯示，點擊右上角的對話圖示即可展開。

💬 AI 對話面板深入介紹

AI 對話面板是你與 Skill 互動的主要介面。理解它的運作方式，對於後續的測試和除錯非常重要。

運作流程

  1. 你在對話面板輸入需求（例如：「幫我回覆這封客戶來信」）
  2. AI 掃描所有可用的 Skills -- 讀取每個 Skill 的 description
  3. AI 判斷是否匹配 -- 如果你的需求符合某個 Skill 的 description，AI 自動載入該 Skill
  4. AI 按照 SKILL.md 的規則執行 -- 依照前置檢查、執行步驟、安全規則產出結果
💡 重要觀念：你不需要手動告訴 AI「使用 email-reply-helper」！AI 會根據 description 的內容自動判斷。這就是為什麼 description 要寫得精準的原因。
⚠️ 如果 AI 沒有觸發 Skill：表示 AI 認為你的需求和 Skill 的 description 不匹配。這時候你可以更明確地描述你的需求，或者直接在對話中提到「用 Email 回覆的方式」來引導 AI。

📁 Step 1：建立 .agent/skills/ 資料夾

Antigravity 的 Skill 路徑為 .agent/skills/，所有 Skill 都必須放在這個目錄下，AI 才能找到它們。

建立目錄的方法

在 Antigravity 的終端機中執行以下指令：

# 建立 Skill 的完整目錄結構
mkdir -p .agent/skills/email-reply-helper/assets

建好後的結構

my-project/ └── .agent/ └── skills/ └── email-reply-helper/ ├── SKILL.md ← 核心：你寫的 Skill 定義 └── assets/ └── email-template.md ← 輔助資源（選用）
⚠️ 路徑陷阱：注意 .agent 開頭有一個點（.），這是隱藏資料夾。放錯位置（例如寫成 agent/skills/ 少了點）AI 就找不到！
💡 同步技巧：如果你在 Part 3 已經在 skills-source/ 中寫好了 SKILL.md，現在只需要把檔案複製到 .agent/skills/ 目錄即可。後續的課程會教你如何用同步腳本自動化這個過程。

✅ Step 2：確認 Skill 結構正確

在開始測試之前，務必確認所有檔案都放在正確的位置。一個小錯誤就可能導致 Skill 無法被 AI 偵測到。

必須確認的清單

  • ☐ .agent/skills/email-reply-helper/ 資料夾存在
  • ☐ SKILL.md 放在 email-reply-helper/ 裡面（不是放在 skills/ 根目錄）
  • ☐ SKILL.md 的 YAML name 值 = 資料夾名稱 = email-reply-helper
  • ☐ assets/ 子資料夾已建立（裡面放了 email-template.md）
  • ☐ 所有檔案都是 UTF-8 編碼（避免中文亂碼）

快速驗證指令

# 列出 Skill 資料夾的所有檔案
ls -la .agent/skills/email-reply-helper/

# 確認 SKILL.md 的前幾行（檢查 YAML 格式）
head -10 .agent/skills/email-reply-helper/SKILL.md
✅ 結構正確！如果以上清單全部通過，恭喜你，Skill 已經部署完成，可以開始測試了！

🔨 Step 3：第一次測試

萬事俱備！現在來測試我們的 Skill 是否能正確運作。在 AI 對話面板中輸入以下範例：

📩 測試輸入：
「我收到一封詢價信，請幫我回覆：

您好，我們公司想詢問貴司 A 型號產品 500 組的報價，希望能在下個月 15 日前交貨，請問可以嗎？

張小明
ABC 科技 採購部」

這是一封典型的詢價信，AI 如果成功載入 Skill，應該要：

  1. 判斷來信類型為「詢價」 -- Step 1 的分析結果
  2. 分析出三個需求：報價、數量（500 組）、交期（下月 15 日）
  3. 按照 SKILL.md 的輸出格式撰寫回覆（主旨行、稱謂、正文、敬語、署名）
  4. 遵守 NEVER 規則（不直接報價、不承諾折扣）
⚠️ 觀察重點：看 AI 的回覆是否「看起來像套了格式」。如果回覆只是一般的對話式回答（沒有固定格式），表示 Skill 可能沒有被觸發。回到 Slide 6 確認路徑是否正確。

✨ 看結果！AI 按照 Skill 規範回覆

如果 Skill 正確觸發，AI 的回覆會包含以下三大區塊，完全按照 SKILL.md 的規則產出：

📊 分析摘要

來信類型：詢價
情緒判斷：中性/友善
重點需求：報價、數量（500 組）、交期（下月 15 日）
這就是 Step 1「分析來信」的產出物。

📧 回覆信件

格式完整：主旨行 → 稱謂 → 正文（感謝 + 逐一回覆 + 後續行動） → 敬語 → 署名
這就是 Step 2「撰寫回覆」的產出物，完全符合輸出格式設定。

📋 執行摘要

來信分類 + 處理方式 + 風險提醒 + 建議後續
附上使用建議和下一步行動方案，讓使用者快速了解 AI 做了什麼。

🔍 驗證 Skill 是否生效的三個指標

檢查項目期望結果如果不對怎麼修
格式一致回覆有明確的段落結構在輸出格式章節加入更具體的範例
規則遵守沒有直接報價或承諾折扣強化 NEVER 規則的描述
語氣正確專業商業語氣，非口語化在 MUST 規則中加入語氣範例
✅ 成功！AI 完全按照 SKILL.md 的規則執行！這就是 Skill 的威力——每次都能穩定地產出品質一致的結果。

😤 進階測試：客訴情境

一個好的 Skill 不只能處理簡單的詢價，還要能自動調整策略來應對不同的情境。讓我們用一封客訴信來測試：

📩 客訴測試輸入：
「你們公司的服務真的讓人很失望！上次訂購的 B 型號產品有三分之一是瑕疵品，聯繫客服兩天了都沒人回覆。如果這個問題不在本週內解決，我們將考慮終止合作關係。」

AI 應該自動調整的回覆策略

項目詢價回覆客訴回覆
情緒判斷中性/友善負面/不滿
回覆類型詢價客訴
語氣調整正式友善同理心 + 解決方案導向
回覆重點感謝來信 + 回覆問題致歉 + 說明處理方案 + 時間承諾
後續行動邀請進一步聯繫指派專人跟進 + 回報時程
💡 為什麼要多種情境測試？同一個 Skill 面對不同類型的來信，AI 會根據 Step 1「分析來信」的結果，在 Step 2 中自動選擇不同的回覆框架。測試多種情境，才能確保你的 Skill 在各種情況下都能穩定運作。
🎯 觀察重點：客訴回覆的第一段應該是「致歉與同理心」，而非「感謝來信」。如果 AI 用詢價的語氣回覆客訴，表示你的執行步驟需要加入更明確的情境分支邏輯。

⚠️ YAML 常見錯誤與修正

如果 Skill 沒有如預期運作，90% 的問題出在 YAML 格式。以下是最常見的五種錯誤和正確寫法：

錯誤類型❌ 錯誤寫法✅ 正確寫法說明
缺少空格 name:value name: value 冒號後面一定要有空格
混用 Tab 用 Tab 鍵縮排 統一使用 2 個空格 YAML 不接受 Tab，只接受空格
忘記 > 符號 description:
　多行文字...		 description: >
　多行文字...		 多行文字要加 >
name 不一致 資料夾 email-reply-helper
name: email_reply_helper		 兩者完全一致：
email-reply-helper		 大小寫、連字號都要相同
全形符號 name：value（全形冒號） name: value（半形冒號） 所有符號必須是半形英文
💡 除錯心法：遇到 Skill 不生效時，先檢查 YAML 格式！用 head -10 SKILL.md 查看前十行，確認 --- 開頭結尾、冒號後有空格、name 和資料夾名一致。這三點解決了 90% 的問題。

🔧 除錯技巧總整理

Skill 寫好不代表一定能跑得好。以下是最常見的四種問題和對應的解決方法，請務必記住：

1

Skill 沒有觸發

現象：AI 用一般對話方式回覆，沒有套用 Skill 的格式。

原因：description 寫得太模糊，AI 無法判斷應該啟用。

解法：在 description 中明確寫出「當使用者 _____ 時，使用此技能」，讓觸發條件一目瞭然。同時檢查路徑是否正確。

2

輸出格式不對

現象：AI 有回覆但格式混亂，和你設定的不一樣。

原因：「輸出格式」章節的描述不夠具體。

解法：在輸出格式章節加入一個完整的範例，讓 AI 看到「正確答案長這樣」。格式越明確，結果越穩定。

3

AI 忽略規則

現象：AI 做了你禁止的事（例如直接報價）。

原因：重要規則沒有用 MUST/NEVER 標記。

解法：將關鍵規則用 MUST 和 NEVER 強調，放在安全規則章節的顯眼位置。AI 對這兩個關鍵字有特殊的注意力。

4

結果不穩定

現象：同樣的輸入，每次產出的品質和格式都不同。

原因：執行步驟描述太籠統，AI 每次走不同的路徑。

解法：增加更多具體的步驟說明，每一步都要有明確的輸入和輸出，減少 AI 自由發揮的空間。

🎯 除錯心法：遇到問題時，先問自己：「如果我是 AI，看到這段指令，我會怎麼做？」如果你自己看了都覺得模糊，AI 當然也會困惑。Skill 的品質 = 指令的明確程度。
