🔨 建立 Skill 1：need-analyzer

mkdir -p my-project/.agent/skills/need-analyzer

SKILL.md 核心內容

---
name: need-analyzer
description: >
  分析客戶需求文件或來信，提取關鍵需求並分類。
  當使用者提供客戶的需求描述、RFP 文件、
  或詢問函時，使用此技能進行需求分析。
---

# 客戶需求分析器

## 執行步驟
### Step 1：需求識別
- 識別所有明確提出的需求項目
- 識別隱含的（未直接說明但可推斷的）需求

### Step 2：需求分類
- 核心需求：客戶明確要求，必須滿足
- 期望需求：客戶希望有，但非必要
- 潛在需求：AI 建議的加分項目

### Step 3：風險標記
- Check if 需求涉及第三方整合 → 標記「需協調」
- Check if 時程低於 2 週 → 標記「時程風險」

### Step 4：產出結構化摘要（此格式供 proposal-drafter 使用）
- MUST 包含核心需求表格（含優先級）
- NEVER 遗漏客戶明確提到的任何需求

🔨 建立 Skill 2：proposal-drafter

---
name: proposal-drafter
description: >
  根據需求分析摘要撰寫客戶提案書草稿。
  當使用者提供需求分析結果或要求撰寫提案書時使用。
---

# 客戶提案撰寫器

## 前置條件
- MUST 確認已有需求分析摘要（來自 need-analyzer 的輸出）
- Check if 使用者直接提供原始需求 → 建議先執行 need-analyzer
- Ask user 提供公司名稱和方案定價範圍

## 執行步驟
### Step 2：撰寫提案架構
依照 @assets/proposal-template.md 的範本結構撰寫

### Step 3：填充內容
- MUST 针對每個核心需求提出具體解決方案
- MUST 包含時程規劃（甘特圖式表格）
- MUST 包含報價概估（以範圍呈現）
- NEVER 承諾未確認的技術細節
- NEVER 使用過多技術術語

🔨 建立 Skill 3：proposal-reviewer

---
name: proposal-reviewer
description: >
  審查提案書草稿的品質，找出問題並改善。
  當使用者提供提案書草稿，要求審查或優化時使用。
---

# 提案審查器

## 審查項目
| 面向 | 檢查重點 |
|------|----------|
| 完整性 | 是否涵蓋所有核心需求？ |
| 說服力 | 是否有明確的價値主張？ |
| 可行性 | 時程和預算是否合理？ |
| 格式 | 是否符合專業商務文件格式？ |

## 執行步驟
- 逐項檢查上表的四個面向
- MUST 列出發現的所有問題（按嚴重程度排序）
- MUST 提供每個問題的具體修改建議
- 自動修正明確的錯誤（錯字、格式不一致）
- Ask user 確認需要人工判斷的重大修改
- NEVER 降低提案書的專業水準

🔗 Skill Chain 三大設計模式

模式一：線性鏈 (Sequential)

A → B → C → 最終輸出

適用：步驟明確、順序固定的流程

範例：需求分析 → 提案撰寫 → 審查提案

模式二：分支鏈 (Branching)

A（分類）→ 條件判斷 → B1 或 B2

適用：根據條件選擇不同處理路徑

範例：分類來信，詢問/客訴走不同 Skill

模式三：匯聚鏈 (Aggregation)

A1 + A2 + A3 → B（整合）→ 最終輸出

適用：需要整合多個來源的資訊

範例：各部門數據 → 整合報告

🎯 建立流程編排 Skill：proposal-workflow

「流程編排 Skill」不自己做任何處理——它的工作是「指揮」其他 Skill 的執行順序。就像餐廳的「領班」：他不炒菜，但他決定哪道菜先做、誰來做。 
---
name: proposal-workflow
description: >
  客戶提案全流程自動化工作鏈。
  當使用者要求「走完提案流程」「從需求到提案」時使用。
  此技能依序呼叫 need-analyzer、proposal-drafter、
  proposal-reviewer 完成完整提案流程。
---

# 客戶提案工作流程

## 執行流程（5個階段）

### Phase 1：需求收集與確認
- Ask user 提供客戶的原始需求描述
- MUST 確認資訊足夠才能繼續

### Phase 2：需求分析（need-analyzer 邏輯）
- 依照 need-analyzer 規則分析需求
- 展示結果 → Ask user 確認

### Phase 3：撰寫提案（proposal-drafter 邏輯）
- Ask user 補充公司優勢和報價範圍
- 依照 proposal-drafter 規則撰寫提案

### Phase 4：審查定稿（proposal-reviewer 邏輯）
- 依照 proposal-reviewer 規則審查
- 自動修正明確問題，人工判斷大改動

### Phase 5：輸出交付
- MUST 產出最終版提案書（完整 Markdown）

## 進度顯示（每個階段都要顯示）
